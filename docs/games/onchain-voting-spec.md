## 1. Зачем (что чиним)

Сегодня: голоса и подсчёт живут на нашем сервере, исход на цепочку переносит один ключ —
резолвер-оператор. Деньги защищены (резолвер выбирает только сторону, при бездействии работает
таймер), но исходу спора приходится верить нам. Три принятых компромисса держатся на этом:
CR-1 (вес можно исказить), CR-2 (тейкдаун ослепляет жюри), CR-3 (бан глушит голос).

После G3b система обязана давать пять свойств:

1. **Вес голоса неподкупен** — живёт в цепочке, наши руки не дотягиваются.
2. **Голос бесплатен и незаглушаем** — подпись сообщения; донести её в цепь может кто угодно.
3. **Исход считает контракт** — резолвер-ключ удалён физически.
4. **Ложный спор дорог, честный — доступен** — депозиты и асимметрия наказаний.
5. **Каждый платит за своё, победитель забирает сам** — клейм-модель сохраняется.

## 2. Два новых кирпича кости

**2.1 Счётчик репутации** — ончейн-аккаунт (PDA) на каждую пару «канал × кошелёк»: хранит очки,
их историю (чекпоинты) и карманы для голосов (слоты). Это второй носитель репутации; оффчейн-журнал
остаётся продуктовой истиной для UI/тиров, оба обязаны совпадать (§10).

**2.2 Программа споров** — контракт: открытие спора, приём голосов-подписей, табло, финализация,
дельты репутации, банк спора с возмещениями. Становится «резолвером» эскроу вместо ключа оператора.

---

## 3. Счётчик репутации (ончейн)

### 3.1 Создание и структура

- Создаётся автоматически при **первом донате** кошелька на канал, в той же транзакции; ренту
  (~копейки) платит донор — разово на канал.
- Поля: `points` (micro-очки, u64; 1 USDC = 1_000_000 micro-очков = 1 очко — курс жёсткий,
  ADR 0007), чекпоинты, слоты голосов, bump.
- **Инструкции передачи НЕ существует** — непередаваемость (§4.3) становится физическим свойством.

### 3.2 Писатели (исчерпывающий список)

| Писатель | Что пишет | Основание |
|---|---|---|
| Инструкция прямого доната | `+= полная сумма` (micro) | деньги необратимо ушли стримеру в той же транзакции |
| **Релиз эскроу** (`claim_streamer` / финализация с исходом to_streamer) | `+= полная сумма задания` | деньги необратимо ушли стримеру. **`fund` НЕ пишет ничего**: заморозка ≠ донат; рефанднутый эскроу не даёт ни очка (иначе clawback-фарм репутации — очки только при релизе, паритет с оффчейн-правилом repEffects) |
| Программа споров (finalize) | `+DISPUTE_WIN_BONUS` / `−DISPUTE_LOSS_PENALTY` оспаривающему (пол на нуле) | вердикт взвешенного голосования |

Больше НИКОГО: ни оператора, ни сервера, ни стримера. Добавление нового писателя (будущая
мини-игра) = апгрейд программы = публичная процедура через таймлок (§9). Тихо печатать
репутацию не может никто, включая нас.

### 3.3 Чекпоинты (снимок веса)

Счётчик хранит append-only список `(слот, значение)` — запись добавляется при каждом изменении.
Вес голоса в споре = значение на последний чекпоинт **≤ момента открытия спора**. Это убивает
накачку веса во время голосования: донаты после открытия спора вес в нём не меняют.
Ренту за рост списка (≈16 байт на чекпоинт, доли цента) платит владелец счётчика — донор
оплачивает собственную историю.

**Компакция:** истории старше слота открытия самого раннего ЖИВОГО спора канала никто уже не
запросит → чекпоинты до этой границы схлопываются в одну базовую запись (permissionless-инструкция
уборки; освободившаяся рента — владельцу счётчика). Аккаунт активного кита не растёт бесконечно.

### 3.4 Слоты голосов (убийца залогов)

В счётчике зарезервировано **K = 4** карманов вида `(dispute_id, commit_hash)`. Голос пишется в
свободный карман СВОЕГО счётчика — новые аккаунты не создаются, предельная стоимость голоса по
хранению = 0. После финализации спора карман освобождается. Кошелёк, голосующий одновременно в
>K спорах ОДНОГО канала, — редкость: для него создаётся временная overflow-запись (~$0.2 залога,
возвратного). Разные каналы не конкурируют (у каждого свой счётчик).

## 4. Жизненный цикл спора

### 4.0 Параметры фиксируются заранее (анти-манипуляция)

Пороги/кворум/окна голосования живут в **ончейн-конфиге канала** (маленький PDA, изменения
подписывает ключ стримера — тот же приём, что аттестация payout). Спор использует параметры,
действовавшие **на момент создания задания**, либо изменение параметров вступает в силу с
задержкой (таймлок ≥ длины полного цикла задания) — иначе стример, предвидя спор, поднял бы
порог голоса до небес и отрезал жюри. Выбор варианта — на реализации (склоняемся к таймлоку
изменений: проще для контракта). ⚠️ Это обязательный замок, без него §8.9 открыта.

### 4.1 Открытие (`open_dispute`)

- Кто: любой кошелёк с весом ≥ `minReputationToDispute` на канале (счётчик, на момент «Готово»),
  кроме самого стримера.
- Оспаривающий вносит **депозит спора** + залог аккаунта спора (возвратный).
- Фиксируется: слот снимка весов (= слот открытия), параметры голосования, окна, банк спора.
- Эскроу переводится в Disputed (замораживается от `resolve_timeout` до вердикта — как сейчас).

### 4.2 Конверты (commit)

- Голос = **подписанное сообщение** (не транзакция): кошелёк подписывает бесплатно, как при входе.
- В конверт-фазе сообщение несёт только `hash(выбор ‖ соль)` — до вскрытия никто (включая нас и
  подателей) не знает, за кого голос → ни копирования лидера, ни выборочной цензуры по стороне.
  (Продажу голоса кооперативному продавцу конверты НЕ закрывают — честно: §8.5.)
- Формат сообщения (анти-переигрывание): домен приложения, id спора, id канала, хэш конверта,
  версия. Подпись из одного спора физически не подойдёт к другому.
- Подача: пачками по ~4–8 голосов на транзакцию, **permissionless** — податель прикладывает
  подписи, контракт проверяет их нативной Ed25519-программой Solana, сверяет вес ≥
  `minWeightToVote` (порог канала, дефолт 1 очко — пыль не голосует), пишет конверт в слот
  счётчика голосующего. Дубль (слот уже занят этим спором) — отказ.
- **Право голоса при коммите проверяется по ТОМУ ЖЕ снимку, что и вес при вскрытии** — чекпоинт
  на слот открытия спора, не текущее значение. Никакого рассинхрона commit/reveal.

### 4.3 Вскрытие (reveal)

- После закрытия конверт-окна голосующий подписывает второе сообщение: `(выбор, соль)`.
  Подаётся так же пачками кем угодно.
- Контракт: сверяет хэш с конвертом, читает вес из чекпоинта на слот открытия спора,
  **капельно прибавляет к табло**. Невскрытый конверт не считается (голос сгорел — стимул
  вскрываться; вскрытие тоже бесплатно).
- **Табло шардировано**: K суб-аккумуляторов (PDA; старт K=16, параметр), вскрытие пишет в шард
  `hash(голосующий) mod K`. Solana сериализует записи в один writable-аккаунт — единое табло
  стало бы горячей точкой, куда 100k вскрытий вставали бы в последовательную очередь (стена не
  по деньгам, а по времени и приоритетным комиссиям). Шарды параллелятся; финализация суммирует
  K шардов — O(K), копеечно. K проверяется нагрузочной репетицией (V3).

### 4.4 Финализация (`finalize`) — permissionless

По истечении окна вскрытия любой вызывает финализацию (копеечная транзакция, возместима из
банка спора). Контракт:

1. читает готовое табло (перебора голосов нет — стоимость не зависит от явки);
2. пишет исход в эскроу (`ToStreamer`/`ToDonor`) — дальше обычный claim;
3. применяет дельты репутации (§7.2);
4. распределяет банк спора (§6);
5. освобождает слоты; возвращает возвратное.

### 4.5 Таблица исходов

Депозит состоит из двух частей: **невозвратная арбитражная пошлина** (покрывает фактические
издержки подачи/финализации, с потолком) и **возвратная залоговая часть**.

| Итог голосования | Деньги | Оспаривающий | Депозит |
|---|---|---|---|
| Большинство весов: «не выполнил» | 100% донору | **+10** очков | залог назад, И пошлина компенсируется за счёт проигравшего — победитель в нуле |
| Большинство весов: «выполнил» | стримеру (97/3) | **−50** очков | **сгорает целиком** (в банк спора) |
| Кворум не собран | стримеру (презумпция §11 спеки игры) | **без штрафа** | залог назад; пошлина — по фактическим издержкам подачи («вызвал суд — оплати автобус суда») |
| Ничья по весу | стримеру (презумпция) | **без штрафа** | как при no-quorum |

Наказывается только **активный проигрыш** (комьюнити явно проголосовало против) — «попробовал
честно, все промолчали» штрафа не несёт. Это уже так в оффчейн-машине; контракт наследует.

## 5. Цена для участников

| Участник | Платит |
|---|---|
| **Голосующий** | **0. Всегда.** Две бесплатные подписи (конверт + вскрытие). SOL на кошельке не нужен |
| Донор | как сейчас: донат/fund + разовая рента счётчика при первом донате на канал |
| Стример | как сейчас: accept/done; claim выигрыша — сам (клейм-модель) |
| Оспаривающий | депозит = невозвратная пошлина (факт. издержки цепи, с потолком) + возвратный залог; залог аккаунта спора (возвратный) |
| Податель пачек | комиссии по факту (~0.1 цента/подпись), **возмещаются из банка спора** с потолком |
| Финализатор | одна транзакция, возместима оттуда же |
| Платформа | ничего обязательного; наш сервис подаёт пачки как удобство и получает возмещение как все |

Арбитражные издержки в итоге ложатся на **проигравшую сторону** (вычитаются из её доли банка/
депозита) — «судебные издержки». Порядок величин: спор на 1 000 голосов ≈ $3–5 сгорающих
комиссий; на 100 000 ≈ $300–500 (и это канал с многомиллионным оборотом донатов).

### 5.1 Спонсорство спора (опционально)

Любой может пополнить банк спора деньгами (покрыть издержки большого жюри). Взнос возвратен
после финализации. Вознаграждение — **только косметика в коже** (бейдж «спонсор», строка в
профиле): ни репутации, ни веса — репутация покупается только донатами стримеру (§8.4).
«Мест» в споре не существует и не продаётся: порог веса — единственный гейт (§8.9-запрет).

## 6. Банк спора

Наполнение: **невозвратная пошлина оспаривающего** (есть ВСЕГДА, включая исход no-quorum —
банк не бывает пуст, см. §8.15) + при активном проигрыше любой стороны арбитражная надбавка из
её доли. Расход: возмещение подателям (по факту, с потолком = фактическая комиссия), возмещение
финализатору, остаток — по исходу (§4.5). Податели работают **в пределах предфинансированного
банка** (сверх бюджета пачки не подаются — защита от осушения); возмещение делает подачу
самоокупаемой для любого курьера: наш сервис не обязателен, рынок подачи открыт.

### 7.2 Экономика

| Параметр | Старт | Комментарий |
|---|---|---|
| `DISPUTE_WIN_BONUS` | +10 очков | как сейчас |
| `DISPUTE_LOSS_PENALTY` | −50 очков (пол на нуле) | ложный спор EV-отрицателен |
| Депозит спора | пошлина (факт. издержки, потолок ~$5–10) + залог (**калибровка**; ориентир: max($1, 5% суммы), потолок $50) | залог сгорает только при активном проигрыше; пошлина невозвратна |
| Кворум | **сублинейный**: `max(1, ⌈2·√(сумма в USDC)⌉)` ($1→2, $100→20, $10 000→200) | линейный «кворум = сумма» защищал недобросовестного стримера сильнее всего на крупных суммах (no-quorum → ему; чем дороже задание, тем недостижимее явка). √-кворум держит жюри досягаемым там, где денег больше всего. Калибровка devnet |
| `minReputationToDispute` | рычаг стримера (есть) | право открыть спор |
| `minWeightToVote` | **новый** рычаг стримера, дефолт 1 очко | право голосовать; заодно отсекает голоса нулевого веса |
| K (слоты голосов) | 4 | одновременные споры одного канала |

### 7.3 Молодой канал (cold-start) и D_max

На канале без распределённой репутации жюри физически нет → любой спор уйдёт по презумпции
таймера. Это максимальный риск ровно тогда, когда стример непроверен. Решение БЕЗ возврата
траста (операторского арбитража «на первые N заданий» НЕ будет — он реинтродуцировал бы
доверие к нам в исход спора, линия CR-2; напряжение названо и принято):

1. **D_max** (whitepaper): потолок суммы задания как функция зрелости жюри канала — например от
   числа независимых кошельков с весом ≥ `minWeightToVote` и суммарного распределённого веса.
   Молодой канал = маленькие задания = ограниченный ущерб, пока жюри не выросло.
2. **Честный UI-варнинг донору** (кожа): «жюри на этом канале ещё не сформировано — спор решит
   таймер-презумпция; лимит задания $X». Никаких заглушек-обманок.
3. Формула D_max — калибровка devnet (§14).

### 7.4 Commit-reveal порог

Whitepaper: конверты обязательны для споров ≥ $50; мелкие можно открытым голосованием (один шаг,
вдвое меньше подписей и подач). v1 реализует конверты как основной режим; упрощённый режим для
мелких — опция реализации (§14).

| # | Атака | Замок |
|---|---|---|
| 8.1 | Накачка веса во время голосования (самодонаты жюри-сибилам) | чекпоинты: вес взят на слот открытия спора |
| 8.2 | Сибил-армия голосов | вес = задоначенные деньги; `minWeightToVote`; тысяча пустых кошельков = вес 0 |
| 8.3 | Clawback: донор оспаривает выполненное | депозит + −50 (≈ $50 донатов) против выигрыша; EV-отрицательно на типовых суммах; крупные — конверты + кворум |
| 8.4 | Фарм +10 спорами на своём канале | локальность: за пределами канала очки не стоят ничего; ВНУТРИ канала они стоят голосового веса — это честно признаём, замок не «бесполезность», а связка: депозит/−50 за проигрыш + circular-bootstrap (жюри своего канала надо сначала вырастить) + §8.14 |
| 8.5 | Копирование лидера / стадность | конверты: счёт и голоса не видны до вскрытия — «голосовать за победителя» нельзя. **Продажу голоса кооперативному продавцу конверты НЕ закрывают**: голосующий всегда может доказать голос покупателю, показав (выбор, соль) против своего ончейн-конверта. Receipt-freeness требует MACI-подобной схемы — v2-исследование (§14). Экономика подкупа как остаточный замок: скупать нужно вес, купленный реальными донатами этого канала, при ограниченном D_max банке. Статус: **принятый остаток v1** |
| 8.6 | Цензура подачи (податель «теряет» голоса) | конверты неразличимы по стороне; подача permissionless + возмещение = у каждой стороны свой курьер; подпись портативна |
| 8.7 | Бан кошелька глушит жюри (бывш. CR-3) | умирает: голос идёт мимо нашего сервиса |
| 8.8 | Тейкдаун текста ослепляет жюри (бывш. CR-2) | текст прибит коммитментом (CR-4) — подменить нельзя; остаточное: UI-видимость (честно фиксируем) |
| 8.9 | Стример крутит параметры перед спором (порог голоса ↑, кворум ↑) | параметры из ончейн-конфига с фиксацией на момент создания задания ИЛИ таймлоком изменений (§4.0, обязательный замок) |
| 8.10 | Спам-споры | депозит + `minReputationToDispute` |
| 8.11 | Невскрытие конвертов (свои вскрыл, чужие «забыл») | вскрытие подаёт кто угодно, вскрывающая подпись у голосующего; невскрытый голос — потеря только его владельца |
| 8.12 | Осушение банка спора микро-пачками | возмещение с потолком по факту + минимальный размер пачки |
| 8.13 | Мы умерли/легли | всё permissionless: подача, финализация, claim; exit-kit |
| 8.14 | **Кит-судья / самонакрутка жюри стримером** (self-донаты своим сибилам за ~3% стоимости веса; жюри без штрафов — голосует безнаказанно) | Вес остаётся **линейным намеренно**: без proof-of-personhood любая сублинейная кривая (√, квадратичное голосование) ЛОМАЕТСЯ дроблением стейка — √a+√b > √(a+b), сибилы делают «сглаживание китов» театром безопасности; линейность — единственная splitting-нейтральная кривая (тот же аргумент, что закрыл A3). Реальные замки — экономические: **D_max** ограничивает банк спора зрелостью канала (скупать жюри дороже, чем банк); публичность после вскрытия — веса и кластеры голосов видны всем, UI флагает концентрацию в связанных кошельках (кожа); самонакрутка платит нам 3% и остаётся в песочнице канала. Статус: **принятый предел псевдонимного взвешенного голосования**, честно задокументирован |
| 8.15 | Гриф «удорожь хостинг»: споры под no-quorum (частичные коммиты, кворум не добрать) — податели сожгли комиссии, банк пуст | невозвратная пошлина депозита покрывает фактические издержки ВСЕГДА (в т.ч. no-quorum); податели не подают сверх банка. Гриф платит за свой же гриф |

Умирают навсегда: D1/D2 (резолвер), CR-1 (искажение веса), CR-3 (бан-глушение). CR-2 сужается
до UI-видимости.

## 9. Эволюция и governance

- **Число** не меняет никто (только два протокольных писателя §3.2).
- **Правила** — апгрейд программ: devnet — наш ключ (стройка); mainnet — мультисиг + таймлок
  (изменение объявляется, вступает через N дней, есть окно выйти); сжигание ключа — когда
  механика стабильна годами. Апгрейд НЕ трогает накопленные данные (чекпоинты append-only) —
  меняются правила «с этого момента», прошлое неприкосновенно (тот же банкинг-принцип, что в журнале).
- **Смысл числа** (тиры, перки, пороги) — кожа, меняется свободно, как сейчас.
- Продуктовые эксперименты с формулами (decay, сезоны) — только оффчейн-метрики поверх журнала;
  вес голоса всегда жёсткий.

---

## 10. Два носителя, одна истина

Журнал (мышцы) остаётся: UI, тиры, лидерборды, экспорт. Счётчики (кость) — вес голоса.
Оба растут из одних событий → обязаны совпадать. Ревизор `verify-export` получает новую
проверку: «журнал == счётчики по каждой паре канал×кошелёк» (читает счётчики из цепи).
Расхождение = красный флаг само по себе.

---

## 11. Интеграция с эскроу и миграция

- В эскроу-программе константа `RESOLVER` заменяется на PDA программы споров: `mark_disputed`/
  `resolve_dispute` вызывает контракт голосования, не человек. Требует редеплоя эскроу —
  совмещаем с возвратом прод-окон (ESC-9) и пином mint (ESC-7) одним релизом.
- Кнопки резолвера в UI и `ESCROW_RESOLVER` из env — удаляются.
- devnet: стенд обнуляется (тестовые данные); mainnet: счётчики честно стартуют с нуля вместе
  с платформой — миграции истории нет и не нужно.

## 12. Регуляторная заметка

Жюри без приза И без штрафа: голосующий не выигрывает и не теряет ни при каком исходе
(+10/−50 — только оспаривающему) → не пари и не «работа судьёй». Цена этой регуляторной чистоты —
безнаказанность кита (§8.14), ограниченная экономикой D_max. Депозит — анти-спам-механика, не
ставка. Спонсорство — возвратное, без вознаграждения деньгами/репутацией. Мастер-переменная (юрисдикция, допуск США) по-прежнему
решается ДО мейннета — жюри над деньгами усиливает её приоритет (спека игры §14).

## 13. Фазы постройки (каждая — со светофором для владельца)

| Фаза | Что | Как проверяешь ты |
|---|---|---|
| **V0** | Ончейн-конфиг канала + счётчики: донат пишет счётчик; ревизор сверяет журнал==счётчики | `verify-export` — новая ✅-строка; донат в UI как раньше |
| **V1** | Программа споров, ОТКРЫТОЕ голосование (без конвертов) на devnet; эскроу резолвится ею; кнопок резолвера больше нет | сценарий спора из manual-testing тремя кошельками — исход без твоего участия |
| **V2** | Конверты + пороги + депозит + банк/возмещения + спонсорство | обновлённые сценарии: голос-подпись, депозит сгорел/вернулся |
| **V3** | Нагрузочная репетиция: синтетический спор на 10k–100k голосов на devnet | отчёт с цифрами (пачки, время, издержки) |
| **V4** | Аудит (счётчики+споры+интеграция), прод-окна, мультисиг+таймлок, mainnet-гейты | итог аудита в audit-map |

## 14. Открытые вопросы (решить по ходу, не блокируют V0–V1)

1. Калибровка: пошлина/залог депозита, коэффициент √-кворума, **формула D_max** (§7.3) — данные devnet.
2. Фиксация параметров: на момент создания задания vs таймлок изменений конфига (§4.0).
3. Упрощённый одношаговый режим для споров < $50 (без конвертов) — делать ли в v1.
4. K слотов голосов (старт 4) и K шардов табло (старт 16) — подтвердить нагрузкой (V3).
5. State compression (сжатое хранение) для споров-миллионников — только если понадобится.
6. **Receipt-freeness (анти-подкуп)**: MACI-подобная схема как v2-исследование — конверты закрывают
   стадность, но не кооперативную продажу голоса (§8.5). Решить после данных о реальных спорах.
7. Bounty-рынок подателей: достаточно ли возмещения по факту или нужен явный плюс.
