# ADR 0003 — Фаза 2: RPC-мост + серверный in-memory store (без БД в окружении)

- **Статус:** принято
- **Дата:** 2026-06-19
- **Контекст:** Фаза 2 должна заменить мок настоящим бэкендом, не трогая экраны (`backend/spec.md`,
  критерии выхода). В текущем окружении **нет Postgres/SQLite/Docker** и нет прав их поставить.

## Решение

1. **Транспорт — типобезопасный RPC-мост, а не россыпь REST-роутов.** Один route-хендлер
   `POST /api/v1/rpc` принимает `{ method, args, identity, failMode }` и диспетчеризует на серверный
   store по белому списку методов. Это «tRPC-lite»: контракт уже задан интерфейсом `DataProvider`,
   поэтому отдельная REST-поверхность из ~20 файлов избыточна. Полноценный REST/SSE можно добавить позже.

2. **Хранилище — серверный in-memory store как стенд-ин под Postgres.** Источник истины — журнал
   событий; репутация считается тем же **общим** движком `lib/reputation.ts` (ADR 0001), что и в моке →
   цифры совпадают по построению (инвариант §4.4). Singleton кэшируется на `globalThis` (переживает
   HMR, шарится между запросами). Целевая схема описана в `backend/schema.sql`; перевод на Postgres =
   заменить внутренности `src/server/store.ts`, не трогая API и экраны.

3. **`bigint` через транспорт.** Деньги — micro-USDC (`bigint`), JSON их не умеет. Кодек
   `src/lib/data/codec.ts` тегирует `bigint → { __bigint }` и обратно; используется обеими сторонами.

4. **Личность — per-request.** `ApiDataProvider` держит `identityKey`/`failMode` локально и шлёт их с
   каждым запросом; сервер применяет их к store перед вызовом. Это сохраняет работу dev-тулбара и
   `/connect` под `api` и ближе к реальной аутентификации (токен на запрос), чем глобальная сессия.

5. **`ApiDataProvider` реализует тот же `DataProvider`.** Выбор по флагу
   `NEXT_PUBLIC_DATA_SOURCE=api`. Экраны не меняются.

## Последствия

**Плюсы:** минимум кода; паритет мок↔api гарантирован общим движком (проверено: donorB=414000 на обоих);
seam доказан — данные идут с сервера по HTTP, состояние серверное; готово к Postgres сменой одного слоя.

**Минусы / честные ограничения:**
- Persistence — in-memory (нет БД в окружении). Перезапуск сервера = сид заново.
- `subscribeOverlay` под `api` — заглушка (нужен SSE/WebSocket — следующий шаг).
- Внешний авто-слой модерации (OpenAI omni-moderation) не подключён (нет ключей) — работает
  упрощённый серверный wordlist/verdict; дедуп по `content_hash` в сторе пока не реализован.
- Глобальный singleton + per-request установка identity/failMode не потокобезопасны при конкуренции —
  допустимо для dev/одного пользователя; в Postgres-версии состояние уходит в БД и проблема исчезает.

## Альтернативы (отклонены)

- **~20 отдельных REST-роутов под §5.** Больше шаблона без выигрыша на этой фазе; RPC-мост покрывает
  весь интерфейс одним файлом.
- **Дублировать логику переходов на сервере «по-настоящему».** Дублирование ~400 строк и риск
  расхождения мок↔api. Переиспользование общего движка + store-класса честнее держит паритет.
- **Поднимать Postgres.** Невозможно в окружении (нет БД и прав установки).
