# ADR 0013 — Лёгкая локальная файловая персистентность (стенд-ин под Postgres)

**Статус:** принято (2026-06-21)
**Контекст:** Фаза 2/3, dev. Связано с ADR 0003 (in-memory стор как стенд-ин под `backend/schema.sql`).

## Проблема

Серверный стор (`src/server/store.ts` → `MockDataProvider`) и SIWS-сессии (`src/server/auth.ts`) жили
только в памяти процесса. Любой перезапуск dev-сервера (а в dev он частый — при изменении кода стора HMR
не пересоздаёт инстанс, нужен рестарт) **стирал всё**: каналы, журнал репутации, донаты, модерацию, а также
сессии — после каждого рестарта приходилось заново подписывать вход. Это ломало любой сквозной тест и
приводило к «потере» уже оплаченной активации канала.

Полноценный Postgres — целевое решение (`backend/schema.sql`), но в этом окружении его нет (Node поставлен
вручную без sudo), а реализация SQL-репозитория над богатым доменным стором — большой и рисковый объём.

## Решение

Лёгкая **файловая персистентность** без БД и нативных модулей (`src/server/persist.ts`):

- Стор сериализуется методом `MockDataProvider.__snapshot()` (Map → entries) и восстанавливается через
  `__restore()`. `bigint` (деньги в micro-USDC) переживает через существующий codec (`__bigint`-тег).
- На старте `getStore()` читает `.data/store.json` (синхронно, один раз) и восстанавливает состояние.
  Сейвер привязывается к стору ВСЕГДА (не только при создании) — иначе после HMR `persistStore()` остался
  бы no-op.
- После мутаций (`route.ts`: whitelist `MUTATING` + `ingest*`/`__reset`) планируется **атомарная** запись
  (tmp + rename) с троттлингом (≤1 запись/250мс, последнее изменение долетает).
- SIWS-сессии аналогично сохраняются в `.data/auth.json` (nonce'ы — нет, они транзиентны, TTL 5м), так что
  вход переживает рестарт.
- `.data/` в `.gitignore` (там приватный текст донатов и токены сессий).

## Последствия

- **Плюс:** данные и вход переживают перезапуск; стор остаётся быстрым in-memory; нулевой риск установки
  (чистый `node:fs`), экраны и API не тронуты.
- **Минус (честно):** весь датасет грузится в память (ок для dev-масштаба); при жёстком `kill` можно
  потерять последние <250мс изменений (троттл); нет конкурентных писателей/транзакций/миграций.
- **Путь на прод:** заменить внутренности `persist.ts`/`store.ts` на репозиторий по `backend/schema.sql`
  (Postgres) + сессии в Redis/Postgres (см. остаток в ADR 0006/0011). Граница (`DataProvider`/RPC) не
  меняется — это и есть смысл шва.

Скрипты сопровождения: `scripts/snapshot-running.ts` (разовый мост: снять текущий стор с работающего
сервера в файл при переходе на персистентность), `scripts/scan-treasury.ts` / `scripts/recover-activation.ts`
(до-зачёт донатов/активации из цепочки — источник истины о деньгах).
