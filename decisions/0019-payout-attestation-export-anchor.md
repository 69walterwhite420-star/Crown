# ADR 0019 — Трастлесность без нового контракта: аттестация payout (H1), публичный экспорт, пруф-якорь

Дата: 2026-07-02. Статус: принято, реализовано (devnet).

## Контекст

Разбор по слоям доверия (`docs/trust-layers.md`) показал три дешёвых слабых места трастлесности,
решаемых БЕЗ нового ончейн-кода и без новых операторских прав:

1. **H1 (открыт с ADR 0011):** payout-адрес канала диктовал сервер. Взломанный/злой сервер подменяет
   адрес — донор подписывает перевод не туда. Единственный оставшийся денежный рычаг мышц.
2. **§4.4 без кнопки:** «любой может пересчитать репутацию» было декларацией — публичной выгрузки
   журнала/конфигов и готового скрипта пересчёта не существовало.
3. **Централизованный слой без следов:** модерация/конфиги управляемы (это фича), но оператор мог
   тихо переписать прошлое (журнал, версию конфига, «этого тейкдауна не было») — снаружи не поймать.

## Решение

Общий принцип: заменить «поверь серверу» на «проверь подпись/хэш» там, где это дёшево. Ни одного
нового контракта, ни одного нового операторского права.

### 1. Аттестация payout (закрывает H1)

- `lib/chain/attestation.ts` (изоморфный: bs58 + tweetnacl): каноническое сообщение
  «owner X платит на payout Y, v1» + ed25519-проверка.
- Владелец подписывает кошельком: при создании канала (`chain-provider.createChannel` — прозрачно)
  или кнопкой «Подписать адрес выплат» в настройках студии (легаси-каналы, `attestPayout` в
  `DataProvider`). Подпись хранится на канале (`Channel.payoutAttestation`, колонка
  `channels.payout_attestation`).
- Проверки (fail-closed): **клиент донора** перед сборкой донат-tx и эскроу-fund
  (`assertPayoutAttested`); **сервер** при создании/дозакреплении и при зачёте (`ingest`,
  chain-режим). Канал без подписи → донаты приостановлены с честным объяснением в UI.
- Остаточное доверие (задокументировано): привязка handle→owner — платформенная. Аттестация
  гарантирует «деньги идут туда, куда сказал ключ владельца», а не «владелец — тот, за кого себя
  выдаёт хэндл».

### 2. Публичный экспорт + независимый пересчёт (§4.4 — кнопка)

- `GET /api/v1/export/channel/[handle]`: канал (с аттестацией) + все версии конфига + журнал +
  лидерборд. Только публичные данные (журнал текстов не содержит, §4.6). Ссылка «Скачать журнал и
  пересчитать» — на странице канала.
- `GET /api/v1/export/anchor`: полный журнал + конфиги + пер-записные хэши лога модерации
  (содержимое инцидентов приватно — наружу только хэши) + текущие дайджесты + последний якорь.
- `scripts/verify-export.ts`: пересчитывает репутацию движком, проверяет подпись payout и дайджесты;
  `--chain` сверяет якорь и последние донаты с devnet. Дельты истории, банкованные прежней формулой
  (до-A3), — предупреждение, не провал: банкинг прошлое не переписывает (§4.4).

### 3. Пруф-якорь (прозрачность централизованного слоя)

- `lib/data/canonical.ts`: `stableStringify` (сортировка ключей, bigint) + `sha256Hex` БЕЗ
  нормализации (hashContent не подходит — lowercase коллапсирует base58-адреса).
- `server/anchor.ts` + тик в `indexer-service`: при изменении состояния и не чаще
  `ANCHOR_INTERVAL_MS` (дефолт 1 ч) публикуется memo-tx
  `{std:"standing-anchor/1", t, n, j: журнал, c: конфиги, m: модерация}`.
- Подписант — env `ANCHOR_SIGNER_KEYPAIR` (путь к keypair.json или inline-массив), платит ТОЛЬКО
  свой газ — никакого ключа над чужими средствами (§4.1). Без ключа якорь тихо выключен
  (аддитивная фича).
- Смысл: цензура остаётся законным правом кожи, но каждое состояние получает несмываемый
  ончейн-отпечаток — «скрывать можно, скрывать незаметно нельзя».

## Что НЕ сделано намеренно

- **Ончейн-регистрация payout** — не нужна: криптоаттестация решает H1 без контракта; ончейн-вариант
  остаётся опцией при G3b.
- **Upgrade authority эскроу-программы** (проблема «один ключ переписывает правила денег») — на
  devnet оставлен как удобство итераций ОСОЗНАННО; перед mainnet: мультисиг + таймлок (окно для
  выхода через permissionless `resolve_timeout`/`claim`), в идеале сжечь после G3b-аудита.
  Записано: audit-map «известно открытым», trust-layers §2.5, ROADMAP (гейт mainnet).

## Последствия

- Денежный путь донат/эскроу-fund больше не доверяет серверу адрес получателя (двойная проверка:
  клиент + ingest). Легаси-канал devnet должен один раз нажать «Подписать адрес выплат».
- Новая зависимость `tweetnacl` (уже была в дереве web3.js) — явно в package.json.
- БД: колонка `channels.payout_attestation` (+ `ALTER TABLE IF NOT EXISTS` миграция).
- Проверено: typecheck/lint/vitest (116) зелёные, `next build` зелёный; смоук на живом сервере —
  экспорт-роуты 200, `verify-export.ts` сходится (единственный ожидаемый провал — неаттестованный
  легаси-канал, что и есть fail-closed поведение).
