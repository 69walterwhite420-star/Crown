# ADR 0021 — Архитектура v3: миграция трастлесс-слоя на канистры ICP («две кости и кожа»)

Дата: 2026-07-02. Статус: принято владельцем (его предложение), архитектура —
`docs/canister-architecture.md`, реализация не начата (фазы M0–M5).

## Контекст

Вариант Б (ADR 0020: Solana-программа счётчиков + голосование пачками подписей) решал
трастлесность, но тащил тяжёлую инженерию, порождённую дороговизной Solana-аккаунтинга:
чекпоинты в PDA, слоты голосов, шардинг табло, рента-залоги, рынок подателей с возмещениями,
и — самое чувствительное — обёртку священного донат-пути в программу (CPI).

Владелец предложил перенести «дорогую» логику на канистру Internet Computer, которая умеет
трастлесс наблюдать за Solana и управлять Solana-контрактами. Факты проверены по официальным
источникам DFINITY (июль 2026): SOL RPC canister в проде (NNS-контроль, консенсус 3-из-N
RPC-провайдеров, полный набор методов вкл. sendTransaction), тресхолд-Ed25519 позволяет
канистре владеть Solana-адресом и подписывать транзакции, asset-канистры хостят фронт с
сертифицированной раздачей и кастомными доменами, тарифы: хранение ~$0.45/GiB/мес, ingress
~$0.000004/КБ, тресхолд-подпись ~$0.036, reverse-gas (пользователь не платит никогда).

## Решение

**Вариант В** заменяет субстрат варианта Б; экономика споров v1.1 переносится 1:1.

1. **Кость №1 — Solana — только деньги.** Донат-путь не меняется вообще (главный выигрыш
   против Б). Единственное изменение: `RESOLVER` эскроу = тресхолд-Ed25519-адрес канистры
   (bounded, ESC-1) — при mainnet-редеплое вместе с ESC-7/ESC-9.
2. **Кость №2 — `core`-канистра ICP**: индексер Solana (SOL RPC, finalized, порт
   extractDonation), журнал репутации в stable memory (порт computePoints; вес на момент =
   свёртка по дате, без чекпоинт-структур), конфиг-параметры каналов (подпись владельца +
   таймлок), споры/голосование (голос = ed25519-подпись кошелька, ingress-копейки платим мы),
   вердикт → тресхолд-подпись → resolve_dispute (durable nonce), certified-экспорт.
   Хэши текстов — да; сами тексты — никогда.
3. **`assets`-канистра** — фронт (SPA; SSR на канистрах невозможен — конверсия на M3);
   `IcpDataProvider` — четвёртая реализация DataProvider (экраны не меняются, ADR 0001).
4. **Кожа — наш сервер, сжимается**: модерация (OpenAI недетерминирован + приватный ключ —
   в канистре невозможен/не нужен), приватные тексты (снимаемость/комплаенс: state канистры
   видят узлы подсети и его не изъять), /ops, косметика, аналитика. Требование владельца:
   модерация и всё не-жёстко-трастлесс остаются централизованными.

## Модель доверия (главная цена)

Кость №2 мягче кости №1: подсеть ICP = 13/34 известных операторов + governance NNS против
тысяч валидаторов Solana. Иерархия: Solana L1 > подсеть ICP > мультисиг > наш сервер.
Размен принят, потому что радиус поражения ограничен существующими замками: деньги не
переезжают; резолвер bounded (только выбор стороны спорного эскроу); `resolve_timeout`
permissionless (смерть ICP-части = деградация до сегодняшней таймер-презумпции). Худший случай
сговора подсети — неверные исходы спорных эскроу, не кража.

## Что умирает из варианта Б

Программа счётчиков и обёртка доната (CPI), чекпоинты/компакция, слоты голосов и рента,
пачки Ed25519-precompile, податели/возмещения как касса на газ (банк спора остаётся только
экономикой: пошлина+залог, «проигравший платит»), шардинг табло. Сохраняются: вся экономика
v1.1, каталог атак (кроме Solana-субстратных пунктов), конверты для споров ≥$50, локальность
и непередаваемость репутации.

## Новые риски (честно, §7 архитектуры)

Молодость Solana-интеграции ICP (durable nonce, IPv6-провайдеры) — M0-прототип до любых
обязательств; второй стек (Rust-канистры) — смягчение: порт НАШЕЙ уже оттестированной логики
с golden-сверкой 1:1; cycles-опека (freezing) — мониторинг в runbook; SPA-фронт; приватность
стейта канистры; двойная бухгалтерия на переходе обязательна (Solana ↔ канистра ↔ сервер).

## Фазы

M0 прототип (индексер+журнал на канистре, сверка) → M1 канистра=канон репутации → M2 споры +
RESOLVER=тресхолд-адрес (редеплой эскроу) → M3 фронт на asset-канистру → M4 governance
контроллеров (мультисиг→SNS/blackhole) → M5 аудит+mainnet-гейты. Деньги (M2) трогаем только
после доказанных M0–M1. Светофоры владельца — архитектура §9.

## Судьба ADR 0020

Не отменён: экономические решения (депозит-пошлина, активный проигрыш, √-кворум, D_max,
линейный вес как splitting-нейтральный, отказ от «додепа») действуют. Заменён только субстрат
(§ «Решение» п.1–2 ADR 0020 → канистра). Вариант Б остаётся документированным fallback, если
M0 покажет непригодность ICP-связки.
