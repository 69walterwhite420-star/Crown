# ADR 0014 — Постоянство в Postgres через PGlite

**Статус:** принято (2026-06-28)
**Контекст:** Фаза 4. Продолжает ADR 0013 (файловый стенд-ин) и закрывает «путь на прод» из него и
ADR 0003 (in-memory стор как стенд-ин под `backend/schema.sql`).

## Проблема

ADR 0013 дал персистентность через JSON-снимок (`.data/store.json`): весь датасет грузится в память,
запись — целым файлом с троттлингом. Этого хватало для базового dev, но это не БД: нет таблиц, типов,
запросов, транзакций, миграций — и нет пути «посмотреть данные как данные». Целевой Postgres
(`backend/schema.sql`) в окружении поднять нельзя — Node стоит вручную без sudo, Docker/демона нет.

Нужен **настоящий Postgres-слой** (реальные таблицы по целевой схеме), но без внешней инфраструктуры,
установки и прав root — чтобы граница (`DataProvider`/RPC) не менялась, а переход на managed-Postgres в
проде свёлся к смене строки подключения.

## Решение

**PGlite** (`@electric-sql/pglite`) — встроенный Postgres, скомпилированный в WASM, работает прямо в
процессе Node. Данные — в каталоге `.data/pg/`. Ни установки, ни Docker, ни sudo, ни сетевого демона.

- `src/server/db.ts` — `getDb()` (кэш на `globalThis`, переживает HMR) + `ensureSchema` создаёт реальные
  таблицы по `backend/schema.sql` (identities, light_profiles, channels, channel_configs, ledger_events,
  donations, messages, channel_blocks, operator_actions, incident_logs, reports + служебная `meta`).
- `src/server/store-db.ts` — двустороннее отображение домен ↔ таблицы: `bigint` (деньги) ↔ `numeric(20,0)`
  через строки; объекты (тиры/конфиг) ↔ `jsonb`; `Iso` ↔ `timestamptz`. `loadStore()` / `saveStore()`
  переносят снимок стора в таблицы и обратно. Источник истины по репутации — `ledger_events` (append-only),
  число всегда пересчитывается общим движком (`lib/reputation.ts`, инвариант §4.4).
- `src/server/store.ts` — на старте грузит из БД; если БД пуста, **одноразово** переносит прежний
  `.data/store.json` (мягкая миграция с ADR 0013) и сохраняет в таблицы. Логика стора по-прежнему работает
  на быстрой in-memory копии; после мутаций коалесцирующий сейвер пишет копию обратно в БД.
- `/dev/db` (+ `/api/dev/db/data`) — операторский просмотр таблиц (выбор таблицы, сортировка). Доступ
  только кошельку-оператору (`resolveToken` → `OPERATOR_ADDRESS`), т.к. там приватный текст донатов.

## Последствия

- **Плюс:** данные живут в настоящих Postgres-таблицах целевой схемы; нулевой риск установки и прав;
  граница `DataProvider`/RPC не тронута; переход на managed-Postgres = смена подключения, экраны не трогаем;
  есть человеческий просмотр данных.
- **Минус (честно):** стор всё ещё держит полную in-memory копию (прямые SQL-чтения без копии — оптимизация
  на потом); `saveStore` переписывает таблицы целиком (инкрементальная запись — на потом); PGlite —
  одно-процессный встроенный движок, не для горизонтального масштабирования.
- **Остаток:** SIWS-сессии всё ещё файловые (ADR 0013) — перенести в БД; в serverless-проде фоновый
  индексер (`indexer-service.ts`) вынести в отдельный воркер/крон (тот же `ingestSignature`).
