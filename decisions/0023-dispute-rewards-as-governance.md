# ADR 0023 — Награды спора: из фикс-констант в редактируемые governance-параметры канала

- Статус: принято (2026-07-06)
- Контекст-решение владельца: сделать величины изменения репутации по спору настраиваемыми стримером;
  ввести правило «нельзя поднять спор, не имея репутации покрыть возможное списание».

## Контекст

Изменение репутации инициатора по исходу спора задания-доната было **протокольной константой** в двух
местах, держащихся в golden-паритете: TS `machine.ts` (`DISPUTE_WIN_BONUS = 10`, `DISPUTE_LOSS_PENALTY =
50`) и Rust-канистра `disputes.rs` (`*_MICRO`). Это фиксировала запись §18.5-8c («экономика споров без
амоунт-настройки»; ради стабильности подписанного сообщения жило мёртвое поле `dMaxMicro`).

В icp-режиме споры chain-задач решает **канистра** (M2), поэтому «сделать награды редактируемыми» =
править Rust трастлесс-ядра, а не только UI.

## Решение

1. **Награды спора — часть governance-параметров канала** (там же, где кворум/окна/пороги, которые уже
   редактируемы подписью владельца + таймлок). `DisputeParams` += `dispute_win_bonus_micro` /
   `dispute_loss_penalty_micro` (Rust `governance.rs` + TS `dispute-params.ts`); дефолты 10/50 = прежние
   константы. Канон-сообщение подписи → **`v: 3`** (парные пины TS↔Rust + `messages.json` обновлены).
   `rep_effects` (обе стороны) параметризован; `arbiter.rs` подаёт действующие значения канала на резолве
   (кворум/окна снапшотятся при открытии — награды читаются на финализации: компромисс без правки
   хранимого стейта спора). Вне icp (mock/api/chain) награды пока фикс-константы `machine.ts`.
2. **Порог открытия спора флорится штрафом** (icp, `arbiter.rs::open_dispute`): эффективный минимум =
   `max(min_reputation_to_dispute_micro, dispute_loss_penalty_micro)`. Нельзя поднять спор, не имея
   репутации покрыть возможное списание — «у кого репутация ниже, чем у него заберут, спорить не может».
3. **UI:** редакторы наград — в Personal Space → **Mini-games → Dispute governance** (icp); туда же сведён
   весь конфиг игр (пороги §10, реестр игр). Вне icp награды показываются read-only.

## Почему это НЕ ломает инвариант §4.5

§4.5 запрещает **оператору** редактировать репутацию (юридический замок «не ценная бумага»). Здесь
величину калибрует **владелец канала** для СВОЕЙ игры — ровно как он уже задаёт кворум/окна/пороги, —
а применяет дельту **протокол по исходу голосования**, детерминированно и перевычислимо. Оператор
по-прежнему не участвует. `DISPUTE_LOST` остаётся единственной отрицательной дельтой. Отменяется лишь
частная запись §18.5-8c про «фикс-награды», не сам замок.

## Последствия

- Светофоры зелёные: `npm run golden` (v:3), `typecheck`, `npm test` (125), `cargo test` (14 + 7 golden —
  паритет с дефолтами), `next build`.
- **Требует редеплоя канистры** (dfx) для живого icp-стенда; после bump `v:` старые подписи параметров
  невалидны (владелец задаёт награды заново подписью). НЕ предлагать заново: депозиты, √-формулы кворума,
  скрытие табло (§18.5-8b/8c остаются в силе).
- Доки: yellow-paper §18.5-8c(3)(4) + LedgerType + §7.3; `docs/canister-architecture.md` (governance-часть).
